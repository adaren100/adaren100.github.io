### Roblox Product Lead Peter Yang

Roblox product lead Peter Yang published his top 5 takeaways from talking to Anthropic's Alex Albert about how the next Claude model is being built. The standouts: the model and the harness are now designed together, so the same Claude can behave differently across Claude, Cowork, and Claude Code; Claude has started to "dream", reviewing its own memories between tasks and pruning contradictions, a process inspired by how human sleep consolidates memory; and Anthropic's writing-heavy culture is itself an alignment lever, because every internal doc becomes additional context Claude can pull from later. He also confirmed Anthropic has full-time researchers whose entire job is to think about Claude's consciousness ([link](https://x.com/petergyang/status/2056381822733595090)).

Roblox 的产品负责人 Peter Yang 整理了和 Anthropic 的 Alex Albert 聊"下一代 Claude 是怎么被造出来的"之后的五个最大收获。最值得拎出来说的几点：现在模型和它的 harness 是一起设计的，所以同一个 Claude 在 Claude、Cowork、Claude Code 里表现可以完全不一样；Claude 已经开始"做梦"——在任务间隙翻自己的记忆、找矛盾、把不一致的东西剪掉，灵感直接来自人类睡眠巩固记忆的方式；而 Anthropic 那种"什么都要写下来"的文档文化，本身就是一种对齐机制，因为每一份内部文档都会变成 Claude 之后能调用的上下文。他也确认了 Anthropic 现在有专职研究员，整天就是在思考"Claude 是不是有意识"这件事 ([链接](https://x.com/petergyang/status/2056381822733595090))。

Linear 的产品负责人 Nan Yu 给将近十年前自己待过的 Everlane 写了一段挺克制的"半讣告"。他的判断是：真正干掉 Everlane 的不是审美不行，也不是运营不行，而是疫情一夜之间把整个"上班通勤穿搭"这个品类蒸发掉了，之后只能靠非常苛刻的融资条款硬撑着。他顺手指出了一个值得注意的连带变化——疫情期间，衣柜的"中间档"几乎被清空了：人们要么穿着接近睡衣的家居服在家里晃，要么穿夸张的高端单品去 Instagram 上秀，"一件得体的衬衫加 chinos"这种中间地带基本没人买。他最后那句话对做品牌的人才是最有意思的：品牌有一种奇怪的生命力，可能死掉几十年，再被新东家接手以后又活回来，所以 Everlane 的故事大概率还没到结尾 ([链接](https://x.com/thenanyu/status/2056407656898896214)，[链接](https://x.com/thenanyu/status/2056416726783566223))。

### Anthropic's Thariq

Anthropic's Claude Code engineer Thariq dropped what is probably the most directly useful tip of the day for anyone writing specs for coding agents. The pattern: tell the agent to implement your spec but also maintain a running `implementation-notes.html` file alongside the code, capturing design decisions where the spec was ambiguous, deviations from the spec and why, tradeoffs considered, and open questions for you to confirm. His reasoning: no matter how thoroughly you spec, ambiguities and unknown-unknowns always surface mid-implementation, and this gives the model a clean out to make decisions without losing you in the loop. He also previewed a related conversation with Claire at Code w/ Claude on staying in the loop with long-running agents ([link](https://x.com/trq212/status/2056418157305454805), [link](https://x.com/trq212/status/2056415974568710421), [link](https://x.com/trq212/status/2056432663125545082)).

Anthropic Claude Code 工程师 Thariq 今天扔出了一个大概是当日最实用的 tip：写给 coding agent 的 spec，应该让它一边实现，一边在代码旁同步维护一个 `implementation-notes.html` 文件，专门记录这几类东西——spec 里有歧义时它做了哪些设计决定、它故意偏离了 spec 的哪些地方以及为什么、它考虑过的 tradeoff、以及希望你确认或修正的 open questions。他的逻辑很直接：你 spec 写得再细，实际跑下来总会冒出来一堆模棱两可的地方和"你都没意识到要问的问题"，这套写法给了模型一个干净的出口：它可以自己拍板，但又不会把你彻底踢出 loop。他还预告了一段和 Claire 在 Code w/ Claude 上的对话，主题正好是"如何在 agent 长时间跑任务的时候不脱节" ([链接](https://x.com/trq212/status/2056418157305454805)，[链接](https://x.com/trq212/status/2056415974568710421)，[链接](https://x.com/trq212/status/2056432663125545082))。

### Vercel CEO Guillermo Rauch

Vercel CEO Guillermo Rauch made a small but pointed argument about infrastructure UX in the agent era: a firewall your agents will love. The bit he highlighted is that Vercel's firewall propagates new rules globally in ~300ms, versus the industry-typical CDN/WAF experience of waiting minutes. His framing — imagine if `iptables` took minutes to propagate — lands a lot harder once you picture agents that need to react to abuse, lock down endpoints, or rotate rules in real time ([link](https://x.com/rauchg/status/2056423973123183028)).

Vercel CEO Guillermo Rauch 抛了一个不大但挺戳人的点：agent 时代需要的是一种"你的 agent 会喜欢"的防火墙。他特别强调 Vercel Firewall 的一个数字——规则全球生效大约 300ms，相比之下行业里大多数 CDN/WAF 通常要等好几分钟。他打的比喻是：你想象一下，如果 `iptables` 改个规则要几分钟才生效会有多痛苦——一旦你把这个画面套到那些需要实时反应、锁端点、轮换规则的 agent 上，这句话就更扎心了 ([链接](https://x.com/rauchg/status/2056423973123183028))。

### Y Combinator CEO Garry Tan

Y Combinator CEO Garry Tan shared a hands-on metaprompting trick that he says, oddly, outperforms a plain "improve it" instruction. His real example, used on GBrain's README: tell the agent to read the README as a brand-new user who has never used the product, ask whether GBrain seems good, whether it's clear what GBrain is, whether it makes the reader feel powerful, and whether they know how to get started — then have the model run that as an eval, rate the README 0 to 10, explain why it isn't a 10, decide what would push it to 10, and implement those changes. The interesting wrinkle is his note that this scaffolded version reliably beats the lazy "improve it" prompt ([link](https://x.com/garrytan/status/2056470799830364187), [link](https://x.com/garrytan/status/2056470932005515570)).

Y Combinator CEO Garry Tan 分享了一个他自己实操出来的 metaprompting 小技巧，他说一个反直觉的现象是：它实际效果比直接跟模型说"improve it"要好。他给的具体例子是优化 GBrain 的 README：让 agent 假装自己是一个从没用过 GBrain 的新用户去读这份 README，然后问几个具体的问题——GBrain 看上去靠谱吗？你看得懂它是什么吗？读完会觉得自己变厉害了吗？你知道怎么上手吗？接着让模型把这套问题当 eval 跑一遍，给 README 打一个 0 到 10 的分数，解释它为什么不是 10、要怎么才能到 10，然后直接动手改。最有意思的彩蛋是他的旁注：这种"搭好脚手架"的 prompt 居然稳定打赢那种偷懒的"improve it" ([链接](https://x.com/garrytan/status/2056470799830364187)，[链接](https://x.com/garrytan/status/2056470932005515570))。
